## Why

Architecture diagrams exist (per-repo `## Architecture diagram` sections and the instance repo's compiled
`docs/architecture.md`) but nothing links to them from the first place anyone looks: the README. A developer
opening a child repo or the instance repo has no way to discover either diagram without already knowing the
Panopticon layout. This adds discoverable links at the top of every relevant README, and gives the freshly
created instance repo a non-broken starting point instead of template self-description that doesn't apply to it.

## What Changes

- Child repo READMEs gain two links at the top, own-repo diagram above the org diagram: `[{repo}
  architecture](docs/architecture.md)` (relative, resolves once merged into the instance) and `[org
  architecture](https://github.com/{instance}/blob/{instance_default_branch}/docs/architecture.md#{repo})`
  (fully-qualified GitHub URL, resolvable immediately, derived the same way as
  `panopticon.org_diagram_link` — config first, live `gh api` fallback, fail loudly rather than guess a
  branch). Written by the `panopticon-doc-generation` skill as part of its normal architecture-overview pass,
  same treatment as the existing diagram-section back-link.
- Instance repo README gains one relative link at the top: `[org architecture](docs/architecture.md)`. No
  links to individual child repos — the org diagram itself already enumerates them.
- `write_org_diagram` gains an explicit empty-state render: when the compiled index has zero repo sections, it
  writes a placeholder `docs/architecture.md` — a hexagon of six `?` nodes with no meaningful edges, and a link
  at the top to `docs/setup-guide.md#4-initialize-a-child-repo` — instead of an empty or minimal document. This
  placeholder is regenerated by the same deterministic render path every time it runs against zero repos, not
  just written once.
- The template repo ships that placeholder `docs/architecture.md` directly (so a brand-new instance repo's
  link is never dead, even before the first `write_org_diagram` run) and its own `README.md` Overview section
  is replaced: the current template self-description ("this is a GitHub template repository...") is
  inappropriate once copied into an instance repo via "Use this template," and there is no reliable event that
  fires on template-based repo creation to swap it dynamically (confirmed: no dedicated GitHub event exists,
  `create`/`push` are documented as unreliable for this case, and this project's own first code touchpoint,
  `sync-from-template.yml`, is `workflow_dispatch`-only and not guaranteed to run promptly after creation).
  The Overview paragraph is therefore replaced with org-agnostic instance-appropriate boilerplate plus the org
  architecture link, followed by a maintainer note (between that text and the logo image) instructing the org
  to personalize the paragraph — accepted tradeoff: template-repo browsers on GitHub see this note too, since
  template and instance cannot show different content without a creation hook that doesn't exist.

## Capabilities

### Modified Capabilities
- `architecture-diagrams`: adds requirements for README-level diagram links (instance and child repo READMEs)
  and for a deterministic empty-state placeholder render of the org diagram, extending the existing
  diagram-navigation requirements to cover README entry points, not just the diagram section itself.

## Impact

- `panopticon/diagrams.py` (`write_org_diagram`): new empty-state branch producing the placeholder content.
- `.agents/skills/panopticon-doc-generation/SKILL.md`: new rule for writing/refreshing the child repo README
  link block during the architecture-overview pass.
- This repo's own `README.md`: Overview section rewritten (template-appropriate today; becomes the
  instance-appropriate seed once copied via "Use this template").
- This repo's own `docs/architecture.md`: new file, the shipped placeholder.
- `docs/setup-guide.md`: no content change required, but becomes a link target from the placeholder diagram.
- No changes to `panopticon/config.py`, `panopticon/org_diagram_link.py`, or any CI workflow — the child
  README org-link derivation reuses `org_diagram_link`'s existing logic/inputs without adding a new script.
